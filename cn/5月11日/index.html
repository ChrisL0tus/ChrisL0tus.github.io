开篇废话

关于汇编语言的利用其实是一门很高深的艺术，古早的程序员利用0和1写程序，后来的程序员用汇编语言写程序，直到高级语言的出现，才有了现在的我们写的程序。
但是我们的程序在被编译的时候还是会被编译器转化成汇编代码来执行，因此汇编仍是我们需要掌握的底层知识，比较不是比赛中的逆向爷的话，谁会怼着字节码看程序。
汇编的一些指令其实并不很容易记得，我们只需记住常见的几个寄存器操作、系统调用、栈操作其实就可以，遇到不认识的再去翻书、搜资料便是，前提是看完一遍《汇编语言》并做过实验。

赛题中的 艺术性 利用书写

在pwn题中与汇编关系最大的无非是不能翻编译的时候需要自己逆向分析，或是有的题目在text段有一些汇编代码不能被解析成函数或者C代码需要自己undefine和define函数，以及有的难一点的shellcode类题目需要手搓精简shellcode的部分，这种题目山海关的师傅写过一篇文章叫做the art of shellcode，https://mp.weixin.qq.com/s/onpGzz2uzSYKf09yvgb3uA，写的很好，非常值得学习。

这里我以之前遇到的一道shellcode的利用，自己编写0x16字节的shellcode实现控制5个寄存器、传7个参数、调用两个函数为例，展示一下高阶shellcode利用中，汇编语言的写法和艺术

先看exp：

题目截图我懒得放了，做题逻辑就是我上面说的“自己编写0x16字节的shellcode实现控制5个寄存器、传6个参数、调用两个函数”，然后再利用basic getshell的汇编就可以解题。
这里我把关键汇编抽取出来分析一下：

首先我们因为题目里给的read限制长度为0x16，非常短，以及程序里没有/bin/sh地址，而且题目没给libc。那么想法便是自己用汇编再实现一个长写入的read，既然写入之后要能执行，那么还需要mprotect函数改内存权限。因此思路就是实现read(0,,)和mprotect(,,0x7)，题目在main结束前把rax到r15这几个寄存器都赋值成了一个很大的数0x9961，而题目给的内存地址是0x9961000
因此有些寄存器是可以加以利用的，有些寄存器则需要自己改变数值。
首先第一条汇编shel edi，12实现的是将edi寄存器中的0x9961变成0x9961000，即目标内存地址。
Mov dx，0x7做到的是将edx寄存器复制为0x7，写dx是为了缩减汇编长度
Mov ax，10是为了控制系统调用号，第一次调用的是mprotect改权限。其中第二个参数（rsi）寄存器我们不对其进行操作，即利用题目里给的0x9961这个大数作为长度，这也是一种比较巧妙的利用
下面read的参数用类似的方法可以实现，当然，寄存器之间的mov操作会比寄存器和数值之间的mov长度要短，因此压缩汇编指令长度可以尽量利用这一点。
当然如果纯这样利用实现read，做不到0x16长度，最后会卡在0x17卡很久，这时候就需要用到一个不怎么常见的指令，cdq，这个指令它实际的作用是把EDX的所有位都设成EAX最高位的值，如此一来可以用前文的eax控制edx，再后文中清空eax，实现系统调用read。就可以实现我们的目的。

关于docker

这算是以往一直想学的一个东西，其实也就是一个基本的杂项技能，近期会玩了几个指令就也算会用了。
docker模板的话其实挺多，主流的几个ctf的我试着写一下

ctf_xinetd：

在目录下起镜像：
docker build -t "<tag_name>" .
把镜像跑起来
docker run -d -p "0.0.0.0:pub_port:9999" -h "<tag_name>" --name="<name>" name
上面似乎有许多name，tag_name这些东西，但实际使用中图方便直接都设置成同一个tag_name就可以了,端口就自己指定
检查docker进程：
docker ps
停止 docker
docker stop <tag_name>
删除 docker
docker rm <tag_name>
删除 docker镜像
docker rmi <tag_name>
很简单吧，关于配置，直接看语神的pn_docker readme就可以了https://github.com/featherL/pwn_docker
